# 設計スキル

## 目的

要件定義ドキュメントをもとに、実装可能なレベルまで設計を落とし込み、設計ドキュメントを作成・更新する。

## 前提知識

- 現状のアプリケーション設計: [current-state.md](../../../docs/design/current-state.md)
- 要件定義: `docs/requirements/` ディレクトリ
- 既存の設計一覧: `docs/design/` ディレクトリ
- DB スキーマ: `supabase/` 配下の SQL、`src/types/index.ts` の型定義
- ルーティング: `src/app/(pages)/`(画面)、`src/app/api/`(Route Handlers)
- フロントエンド構成: `src/app/(pages)/`、`src/components/`

## 手順

### 1. 要件の確認

- 対象の要件定義ドキュメント(`docs/requirements/`)を読む
- 未決事項がある場合はユーザーに確認する
- 要件が曖昧なら `/requirement` で先に整理することを提案する

### 2. 現状の把握

- `docs/design/current-state.md` で全体アーキテクチャを確認する
- 影響を受ける既存コードを読み、現在の実装を把握する
- DB スキーマ、API、フロント構成を必要に応じて確認する

### 3. 設計

以下の観点で設計する。要件に応じて不要な項目はスキップする。

- **DB設計**: テーブル追加・変更、カラム定義、インデックス、マイグレーション方針(`/db-change` の流儀に従う)
- **API設計**: エンドポイント、リクエスト/レスポンス、認証・認可
- **フロント設計**: 画面構成、コンポーネント分割、状態管理、ルーティング
- **インフラ設計**: Vercel・Supabase の設定変更(環境変数、RLS ポリシーなど)
- **影響範囲**: 既存機能への影響、マイグレーション手順

### 4. ドキュメント作成

以下のフォーマットで `docs/design/` に作成する。ファイル名は `YYYYMMDD-機能名.md`。

```markdown
# [機能名] 設計書

## 概要
- 対応する要件定義: `docs/requirements/YYYYMMDD-xxx.md`
- 作成日:
- ステータス: Draft / Review / Approved

## アーキテクチャ概要
変更の全体像を簡潔に記述する。

## DB設計
### 新規テーブル / 変更テーブル
| カラム名 | 型 | 制約 | 説明 |
|---------|---|------|------|

### マイグレーション方針

## API設計
### [METHOD] /api/xxx
- リクエスト:
- レスポンス:
- 認証:
- エラーケース:

## フロント設計
### 画面構成
### コンポーネント分割
### 状態管理

## インフラ設計

## 影響範囲
- 既存機能への影響:
- 移行手順:

## 未決事項
```

### 5. 次ステップへの誘導

- 設計のレビューが必要なら `/design_review` へ
- 設計が確定したら `/implementation` で実装へ

## 注意事項

- 設計書は実装可能なレベルまで具体化する(テーブル名・カラム名・ファイルパスまで書く)
- 既存の設計(`current-state.md`)と矛盾する場合は、その旨と移行方針を明記する
- 実装が完了して取り込まれたら、`docs/design/current-state.md` に反映して常に現状と一致させる
- 要件にない機能を設計で勝手に追加しない
