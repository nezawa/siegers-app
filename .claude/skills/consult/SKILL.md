# 相談スキル

## 目的

技術的な問題・障害・方針の迷いについて対話形式で整理し、解決策の選択肢とトレードオフを提示して意思決定をサポートする。コードを書くフェーズの前段として、「何が問題か・どう対処するか」を明確にする。

## 前提知識

- 共通規約: `CLAUDE.md` / `AGENTS.md`(Next.js のドキュメント参照ルール)
- 現状の要件: `docs/requirements/current-state.md`
- 現状の設計: `docs/design/current-state.md`

## 手順

### 1. 状況のヒアリング

以下を確認する。一度に全部聞かず、相手の回答に応じて深掘りする。

- **何が起きているか**: エラー内容、バグの症状、迷っている判断
- **背景**: いつから・どのタイミングで発生したか、影響を受けているユーザーや機能
- **現時点の理解**: 原因の仮説、すでに試したこと
- **制約**: 時間・リリース期限・変更できない箇所

### 2. 原因・影響の調査

必要に応じてコードベースを調べ、問題を構造化する。

- 関連するコード・モデル・フローを確認する
- 影響範囲(どのファイル・機能・ユーザーに波及するか)を整理する
- ログ・エラーメッセージがあれば読み解く(既知のエラーは `/troubleshoot` を先に参照)

### 3. 解決策の列挙と比較

複数の選択肢をトレードオフとともに提示する。

```
## 解決策の比較

### 案A: [方針名]
- 概要:
- メリット:
- デメリット・リスク:
- 工数感:

### 案B: [方針名]
- 概要:
- メリット:
- デメリット・リスク:
- 工数感:

### 推奨案: A / B / その他
- 理由:
```

### 4. 方針の決定

ユーザーが選択・決定できるよう整理する。

- 迷っている場合は、判断軸(速さ・安全性・保守性など)を確認して絞り込む
- 決定したら内容を簡潔にまとめる

### 5. 次ステップへの誘導

決定内容に応じて次のスキルへつなぐ。

| 決定内容 | 次のスキル |
|---------|-----------|
| 新機能・仕様変更を伴う | `/requirement` |
| 設計の見直しが必要 | `/design` |
| すぐに実装できる小さな修正 | `/implementation` |
| レビューが必要 | `/review` |

## 相談の例

- バグ報告を受けたが影響範囲が広く、対応方針が決まらない
- 複数の実装方法があり、どれを選ぶべきか判断したい
- 「こういう設計にしようと思うが、問題ないか」を壁打ちしたい
- 顧客対応として今すぐできることと、根本対応に分けて整理したい
