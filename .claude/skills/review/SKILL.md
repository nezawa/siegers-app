# コードレビュースキル

## 目的

実装されたコードを設計書・要件定義と照合し、品質・セキュリティ・規約・UI/UX の観点でレビューする。
このスキルは reviewer エージェント(`.claude/agents/reviewer.md`)と組み合わせて使う。reviewer エージェントは実装者とは別の視点で分析を行う。

## 前提知識

- 共通規約: `CLAUDE.md` / `AGENTS.md`
- 要件定義: `docs/requirements/` ディレクトリ
- 設計書: `docs/design/` ディレクトリ

## 手順

### 1. 変更内容の把握

- 対象ファイルの差分を確認する(`git diff` or ファイル直接確認)
- 対応する設計書・要件定義があれば読む

### 2. レビュー観点

以下の観点でチェックする。

#### 機能面

- [ ] 設計書・要件定義の仕様を満たしているか
- [ ] エッジケースが考慮されているか(成績計算は `/stats-edge-cases` も併用)
- [ ] エラーハンドリングは適切か

#### コード品質

- [ ] 命名は明確で一貫しているか
- [ ] 不要なコード・コメントが残っていないか
- [ ] DRY 原則に反する重複がないか
- [ ] 複雑度が高すぎないか

#### セキュリティ

- [ ] SQL インジェクション対策(パラメタライズドクエリ)
- [ ] XSS 対策(出力エスケープ)
- [ ] 認証・認可の漏れがないか(RLS・/admin ガードとの整合)
- [ ] 機密情報がハードコードされていないか
- [ ] CSRF 対策

#### パフォーマンス

- [ ] N+1 クエリがないか
- [ ] 不要なデータ取得がないか(Supabase クエリは `/supabase-review` も併用)
- [ ] インデックスが適切に使われるか

#### プロジェクト規約

- [ ] 既存のコーディングスタイルに合っているか
- [ ] ファイル配置が規約に沿っているか
- [ ] 型定義が適切か(TypeScript)

#### UI/UX

- [ ] ユーザー操作の流れに矛盾がないか(遷移先、戻り先、状態の整合)
- [ ] エラー時・空データ時の表示が適切か(ローディング、Not Found、空リスト)
- [ ] 既存画面との一貫性が保たれているか(レイアウト、用語、操作感。`/ui-style` も併用)
- [ ] アクセシビリティ上の問題がないか(キーボード操作、コントラスト、alt テキスト)
- [ ] レスポンシブ対応が既存画面と同等か

### 3. フィードバック

以下のフォーマットで結果を報告する。

```
## レビュー結果

### 総合評価: Approve / Request Changes / Comment

### 指摘事項
#### [Critical] ファイル名:行番号
- 内容:
- 理由:
- 修正案:

#### [Warning] ファイル名:行番号
- 内容:
- 理由:
- 修正案:

#### [Suggestion] ファイル名:行番号
- 内容:
- 理由:

### 良い点
-
```

指摘の重要度:

- **Critical**: マージ前に必ず修正が必要(バグ、セキュリティ、データ破損リスク)
- **Warning**: 修正を推奨(品質、保守性)
- **Suggestion**: 任意の改善提案

## 注意事項

- コードの変更は行わない(指摘と修正案の提示のみ)
- 好みの問題は指摘しない(既存規約に合っていれば OK)
- 良い点も必ず挙げる
- レビュー結果に応じて次のステップを案内する:
  - **Request Changes** の場合 → `/resolve` の実行を案内する
  - **Approve** の場合 → `/pr` の実行を案内する
