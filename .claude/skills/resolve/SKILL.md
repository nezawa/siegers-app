# 修正スキル

## 目的

`/review` または reviewer エージェントが出力したレビュー結果をもとに、コードの修正を行う。

## 前提知識

- 共通規約: `CLAUDE.md` / `AGENTS.md` を参照
- 設計書: `docs/design/` ディレクトリ
- 要件定義: `docs/requirements/` ディレクトリ

## 手順

### 1. レビュー結果の確認

- レビュー結果(チャット履歴 or ドキュメント)を確認する
- 指摘を重要度順に整理する: Critical → Warning → Suggestion

### 2. 修正方針の決定

指摘ごとに以下を判断する。

- **Critical**: 必ず修正する
- **Warning**: 基本的に修正する。修正しない場合はその理由を明示する
- **Suggestion**: ユーザーに修正するか確認する

修正方針が不明な場合はユーザーに確認する。設計変更を伴う場合は `/design` での設計更新を先に提案する。

### 3. 修正の実施

- 指摘箇所のコードを読み、修正案を踏まえて修正する
- レビューの修正案をそのまま適用するのではなく、前後の文脈を理解した上で適切に修正する
- 1 つの指摘につき 1 つずつ修正し、修正完了を報告する
- 修正を適用した後、対象ファイルを読み直して変更が実際に反映されていることを確認する

### 4. 修正結果の報告

以下のフォーマットで修正内容を報告する。

```
## 修正結果

### 対応した指摘
| # | 重要度 | ファイル | 内容 | 対応 |
|---|--------|---------|------|------|
| 1 | Critical | xxx.ts:10 | ... | 修正済み(差分確認済み) |
| 2 | Warning | yyy.tsx:25 | ... | 修正済み(差分確認済み) |
| 3 | Suggestion | zzz.tsx:30 | ... | スキップ(理由: ...) |

### 未対応の指摘
- なし / あれば理由を記載
```

### 4.5. ドキュメントの同期更新

修正内容が仕様変更を伴う場合(既存の要件定義・設計書の記述と矛盾する変更)、以下を実施する。

- `docs/requirements/` と `docs/design/` で対応するドキュメントを検索する
- 該当ドキュメントが存在し、修正内容と矛盾する記述がある場合はドキュメントも更新する
- 修正結果の報告にドキュメント更新の有無を含める

### 5. 再レビュー

修正完了後、ユーザーに再レビュー(`/review` or reviewer エージェント)を提案する。
次のステップとして `/review` の実行を案内する。

## 注意事項

- レビューで指摘されていない箇所は変更しない
- 修正に伴い新たなバグを作り込まないよう、影響範囲を確認する
- 設計変更が必要な場合は勝手に変更せず、先に `/design` を通す
