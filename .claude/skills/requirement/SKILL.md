# 要件定義スキル

## 目的

ユーザーから要件を聞き取り、不明・不明瞭な部分を対話で補完しながら、構造化された要件定義ドキュメントを作成する。

## 前提知識

- 現状のアプリケーション要件: [current-state.md](../../../docs/requirements/current-state.md)
- 既存の要件定義一覧: `docs/requirements/` ディレクトリ

## 手順

### 0. ブランチ確認

- 大きめの機能追加・変更は、`feature/` ブランチの作成を提案する
- 小さな変更はこのリポジトリの普段の運用どおり `main` 直コミットでよい

### 1. 現状把握

- `docs/requirements/current-state.md` を読み、現在のアプリケーション全体像を把握する
- 既存の要件定義ドキュメントがあれば確認し、重複や矛盾がないか把握する

### 2. ヒアリング

ユーザーに以下を確認する。不明な点は必ず質問して補完する。

- **目的**: なぜこの機能・変更が必要か
- **対象ユーザー**: 誰が使うのか
- **機能要件**: 何ができるようになるのか(入力・処理・出力)
- **非機能要件**: パフォーマンス、セキュリティ、可用性の制約
- **制約・前提条件**: 技術的制約、ビジネス上の制約
- **受け入れ条件**: 完了の判断基準

質問は一度に大量に投げず、ユーザーの回答に応じて段階的に深掘りする。

### 3. ドキュメント作成

以下のフォーマットで `docs/requirements/` に作成する。

```markdown
# [機能名] 要件定義

## 概要
- 目的:
- 対象ユーザー:
- 作成日:
- ステータス: Draft / Review / Approved

## 背景・課題

## 機能要件
### FR-1: [要件名]
- 説明:
- 入力:
- 処理:
- 出力:
- 優先度: Must / Should / Could

## 非機能要件
### NFR-1: [要件名]
- 説明:
- 基準:

## 制約・前提条件

## 受け入れ条件
- [ ] AC-1:
- [ ] AC-2:

## 未決事項
```

ファイル名は `YYYYMMDD-機能名.md` とする(例: `20260612-asset-upload-improvement.md`)。

### 4. レビュー

作成したドキュメントをユーザーに提示し、過不足がないか確認する。フィードバックがあれば修正する。
ユーザーの承認後、次のステップとして `/design` の実行を案内する。

## 注意事項

- 曖昧な要件をそのまま残さない。不明点は `## 未決事項` に明記する
- 既存機能との影響範囲を必ず確認する
- ドキュメントは日本語で作成する
